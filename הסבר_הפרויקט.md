# הסבר מלא — ספריית ה-Validation ב-Java

> הסבר זה נכתב לכל מי שלא היה בקורס ורוצה להבין מה נבנה כאן, מדוע, ואיך.

---

## מה בנינו בכלל?

בנינו **ספריית Java** שמאפשרת למפתחים לבדוק האם פרטי משתמש חדש (שם, אימייל, סיסמה, גיל) עומדים בכללים שהם מגדירים.

דמיינו שאתם בונים אתר הרשמה. אתם רוצים לוודא:
- האימייל מסתיים ב-`il`
- הסיסמה ארוכה מ-8 תווים
- הגיל מעל 18

במקום לכתוב בדיקות ידניות שוב ושוב, הספרייה שלנו מאפשרת **להרכיב כללים** בצורה נקייה ואלגנטית:

```java
UserValidation rule = UserValidation.emailEndsWithIL()
                          .and(UserValidation.passwordLengthBiggerThan8())
                          .and(UserValidation.ageBiggerThan18());

ValidationResult result = rule.apply(user);
```

---

## תבניות עיצוב — מה זה בכלל?

**תבנית עיצוב (Design Pattern)** היא פתרון כללי שחוזר על עצמו לבעיה נפוצה בתכנות.  
זה לא קוד שמעתיקים — זה **רעיון** על איך לארגן קוד.  
כמו מתכון: "כדי לפתור בעיה מסוג X, כדאי לבנות את הקוד בצורה Y."

בפרויקט זה מימשנו שלוש תבניות עיצוב קלאסיות:

---

## תבנית 1: Combinator (משלב)

זוהי התבנית המרכזית של הפרויקט (**40 נקודות**).

### הרעיון

כותבים כללים קטנים ופשוטים, ואז **מחברים** אותם יחד לכלל גדול ומורכב.  
כמו לגו: כל חלק פשוט בפני עצמו, אבל אפשר לחבר אותם לכל צורה שרוצים.

### הבעיה שהיא פותרת

בלי התבנית הזו, כל פעם שמוסיפים תנאי חדש צריך לשנות קוד קיים.  
עם ה-Combinator, כל כלל הוא עצמאי ואפשר לשלב אותם בצורות שונות בלי לגעת בקוד הישן.

### המימוש — interface `UserValidation`

```java
@FunctionalInterface
public interface UserValidation extends Function<User, ValidationResult> {
    ...
}
```

כל `UserValidation` הוא **פונקציה**: מקבלת `User`, מחזירה `ValidationResult`.

#### כלים לשילוב (instance methods):

| מתודה | מה היא עושה |
|-------|------------|
| `and(other)` | שני הכללים חייבים לעבור |
| `or(other)` | מספיק שאחד יעבור |
| `xor(other)` | בדיוק אחד חייב לעבור (לא שניהם, לא אף אחד) |
| `all(v1, v2, ...)` | כל הכללים חייבים לעבור (varargs) |
| `none(v1, v2, ...)` | אף כלל לא צריך לעבור (varargs) |

#### כלים מובנים (static methods — ללא פרמטרים):

```java
UserValidation.emailEndsWithIL()               // האימייל מסתיים ב-"il"
UserValidation.emailLengthBiggerThan10()        // אורך האימייל > 10
UserValidation.passwordLengthBiggerThan8()      // אורך הסיסמה > 8
UserValidation.passwordIncludesLettersNumbersOnly() // סיסמה: רק אותיות וספרות
UserValidation.passwordIncludesDollarSign()     // הסיסמה מכילה '$'
UserValidation.passwordIsDifferentFromUsername() // הסיסמה שונה משם המשתמש
UserValidation.ageBiggerThan18()               // גיל > 18
UserValidation.usernameLengthBiggerThan8()      // אורך שם המשתמש > 8
```

#### דוגמה לשימוש (מהמטלה):

```java
User user = new User("admin", "admin@#yzw.co.il", "abc123", 34);

UserValidation v1 = UserValidation.emailLengthBiggerThan10();  // 17 > 10 ✓
UserValidation v2 = UserValidation.emailEndsWithIL();          // מסתיים ב-"il" ✓

ValidationResult result = v1.and(v2).apply(user);

// result.isValid() == true → "User is valid"
```

## תבנית 2: Factory Method (שיטת מפעל)

**10 נקודות**

### הרעיון

במקום שהקוד הקורא יחליט איזה סוג אובייקט ליצור, הוא פשוט **מבקש** אובייקט לפי שם — ו**המפעל** מחליט מה ליצור.

### הבעיה שהיא פותרת

נניח שיש לנו שלושה סוגי משתמשים: Basic, Premium, Platinum.  
בלי Factory, הקוד יראה כך:

```java
// בלי Factory — מכוער ושביר
if (type.equals("basic"))    user = new BasicUser(...);
if (type.equals("premium"))  user = new PremiumUser(...);
if (type.equals("platinum")) user = new PlatinumUser(...);
```

זה מופיע בכל מקום בקוד. אם מוסיפים סוג חדש — צריך לחפש ולשנות בעשרות מקומות.

עם Factory:

```java
// עם Factory — נקי ומרוכז
User user = UserFactory.create("premium", "john", "john@test.co.il", "pass123!", 25);
```

הלוגיקה של "מי יוצר מה" מרוכזת במקום אחד בלבד.

### המימוש — `UserFactory`

```java
public class UserFactory {
    public static User create(String type, String username, String email,
                              String password, int age) {
        switch (type.toLowerCase()) {
            case "basic":    return new BasicUser(username, email, password, age);
            case "premium":  return new PremiumUser(username, email, password, age);
            case "platinum": return new PlatinumUser(username, email, password, age);
            default: throw new IllegalArgumentException("Unknown user type: " + type);
        }
    }
}
```

### היררכיית הירושה

```
User (מחלקת בסיס)
 ├── BasicUser
 ├── PremiumUser
 └── PlatinumUser
```

כל תת-מחלקה מפעילה את הקונסטרקטור של `User` דרך `super(...)`.

## תבנית 3: Template Method (שיטת תבנית)

**10 נקודות**

### הרעיון

**שלד האלגוריתם קבוע** — אבל **צעד אחד** (הצעד המשתנה) מושאר פתוח לקורא.

### הבעיה שהיא פותרת

נניח שאתם כותבים מיון. לוגיקת המיון (bubble sort, quick sort וכו') תמיד אותה דבר.  
מה שמשתנה הוא: **איך משווים שני פריטים**.  
אולי פעם ממיינים לפי שם, פעם לפי גיל, פעם לפי אימייל.

בלי Template Method — צריך לכתוב גרסת מיון שונה לכל מקרה.  
עם Template Method — שלד המיון קבוע, רק פונקציית ההשוואה מוחלפת.

### המימוש — `UserUtils.sort`

```java
public static void sort(User[] users, Comparator<User> comparator) {
    int n = users.length;
    for (int i = 0; i < n - 1; i++) {
        for (int j = 0; j < n - i - 1; j++) {
            // ← כאן ה"וו" (hook): ה-Comparator מחליט מי גדול ממי
            if (comparator.compare(users[j], users[j + 1]) > 0) {
                User temp = users[j];
                users[j]     = users[j + 1];
                users[j + 1] = temp;
            }
        }
    }
}
```

### שימוש לדוגמה

```java
User[] users = { userA, userB, userC };

// מיון לפי גיל (סדר עולה)
UserUtils.sort(users, (a, b) -> Integer.compare(a.getAge(), b.getAge()));

// מיון לפי שם משתמש (סדר אלפביתי)
UserUtils.sort(users, (a, b) -> a.getUsername().compareTo(b.getUsername()));
```

## ממשקי התוצאה — `ValidationResult`, `Valid`, `Invalid`

### `ValidationResult` — interface

```java
public interface ValidationResult {
    boolean isValid();
    Optional<String> getReason();
}
```

זה **ממשק** (interface) — הגדרה של מה שכל תוצאת ולידציה חייבת לספק.  
`Optional<String>` הוא מעטפת ל-String שיכולה להיות ריקה — כך נמנע מ-NullPointerException.

### `Valid` — תוצאה חיובית

```java
public class Valid implements ValidationResult {
    public boolean isValid()            { return true; }
    public Optional<String> getReason() { return Optional.empty(); }
}
```

עברנו את הבדיקה → אין סיבה לכישלון → `Optional` ריק.

### `Invalid` — תוצאה שלילית

```java
public class Invalid implements ValidationResult {
    private String reason;

    public Invalid(String reason)       { setReason(reason); }
    public boolean isValid()            { return false; }
    public Optional<String> getReason() { return Optional.of(reason); }
    public void setReason(String r)     { this.reason = r; }
}
```

נכשלנו → יש סיבה → `Optional` מכיל אותה.

## שימו לב לכללי הקוד שנדרשו

### 1. קונסטרקטור דרך Setters
הקונסטרקטור **לא מציב ערכים ישירות לשדות** — הוא קורא ל-setters:
```java
// ✗ שגוי
public User(String username, ...) { this.username = username; }

// ✓ נכון
public User(String username, ...) { setUsername(username); }
```

### 2. סוג המשתנה חייב להיות הממשק
```java
// ✗ שגוי — הסוג הוא המחלקה הקונקרטית
Valid result = new Valid();

// ✓ נכון — הסוג הוא הממשק
ValidationResult result = new Valid();
```

### 3. Javadoc על הכל
כל מחלקה, ממשק, מתודה ושדה — מוסברים ב-`/** ... */`.

### 4. הערות `//` בתוך הקוד
בכל מקום שיש לוגיקה לא-טריוויאלית — יש הסבר.

## סיכום — הקשר בין כל החלקים

```
          UserValidationV2Demo.java
                    │
                    │ creates
                    ▼
                  User ◄──── UserFactory (Factory Method)
                    │
                    │ passed to
                    ▼
             UserValidation ◄──── Combinator: and / or / xor / all / none
          (Function<User, ValidationResult>)
                    │
                    │ returns
                    ▼
           ValidationResult
           ┌────────┴────────┐
         Valid            Invalid
                        (+ reason)

     UserUtils.sort ◄──── Template Method (Comparator = hook)
```

הכל עובד יחד:
1. `UserFactory` יוצר את סוג המשתמש הנכון.
2. `UserValidation` בונה את הכלל המורכב דרך combinator.
3. הכלל מוחל על ה-`User` ומחזיר `Valid` או `Invalid`.
4. `UserUtils.sort` מאפשר מיון גמיש ע"י החלפת לוגיקת ההשוואה בלבד.
