1מה המטרה של המטלה?

דמיינו שאתם בונים אתר שבו משתמשים חדשים נרשמים. בזמן ההרשמה אתם רוצים לוודא שהאימייל שהוזן נראה תקין, שהסיסמה חזקה מספיק, שהמשתמש מעל גיל 18 וכן הלאה.

המטלה מבקשת לבנות ספריית Java בשם validation.jar שתבצע בדיוק את זה — בצורה גנרית, גמישה, וניתנת לשימוש חוזר על ידי כל מפתח.

מה ספריית validation מאפשרת לעשות?

המפתח שמשתמש בספרייה יכול לכתוב קוד כזה:

UserValidation rule = UserValidation.emailEndsWithIL()
                                   .and(UserValidation.passwordLengthBiggerThan8())
                                   .and(UserValidation.ageBiggerThan18());

ValidationResult result = rule.apply(user);

if (result.isValid()) {
    System.out.println("המשתמש עבר את כל הבדיקות");
} else {
    System.out.println("בעיה: " + result.getReason().get());
}

2מה זה תבנית עיצוב?

תבנית עיצוב (Design Pattern) היא פתרון כללי ומוכח לבעיה שחוזרת שוב ושוב בתכנות. זה לא קוד שמעתיקים — זה רעיון על איך לארגן קוד.

אפשר לחשוב על זה כמו מתכון בבישול: אם אתם יודעים להכין "רוטב בסיס", תוכלו להשתמש בו בהרבה מנות שונות. כך גם תבניות עיצוב — הן מתכונים לפתרון בעיות נפוצות בתכנות.

💡
מאיפה זה בא?
הספר המפורסם ביותר בנושא הוא "Design Patterns: Elements of Reusable Object-Oriented Software" מ-1994, שנכתב על ידי ארבעה מחברים המכונים "Gang of Four". הספר מתאר 23 תבניות עיצוב קלאסיות. שלוש מהן נמצאות בפרויקט שלנו.

3תבנית Combinator — לב הפרויקט

COMBINATOR PATTERN

משלב — בונים מהקטן לגדול

כותבים כללים קטנים ופשוטים, ואז מחברים אותם לכלל גדול ומורכב.

ניקוד בסיכון: 40 נקודות

הבעיה שהתבנית פותרת

נניח שיש לנו 8 כללי בדיקה שונים. איך נבדוק שמשתמש עובר כמה מהם? הדרך הנאיבית היא לכתוב if מקוננים: 

// ✗ גישה בעייתית — קשה לתחזוקה, לא גמיש
if (user.getEmail().endsWith("il")) {
    if (user.getPassword().length() > 8) {
        if (user.getAge() > 18) {
            // ...ועוד if...
        }
    }
}

הבעיה: כל פעם שרוצים שילוב אחר של כללים, צריך לכתוב קוד חדש. אי אפשר לשמור כלל ב-variable ולהעביר אותו לפונקציות.

הגישה של Combinator

כל כלל בדיקה הוא אובייקט עצמאי — פונקציה שמקבלת משתמש ומחזירה תוצאה. אפשר לאחסן כלל ב-variable, להעביר אותו לפונקציות, ולשלב עם כללים אחרים: 

// ✓ גישת Combinator — נקי וגמיש לחלוטין

// שמירת כלל ב-variable
UserValidation emailRule = UserValidation.emailEndsWithIL();

// שילוב כמה כללים עם and
UserValidation combined = UserValidation.emailEndsWithIL()
    .and(UserValidation.passwordLengthBiggerThan8())
    .and(UserValidation.ageBiggerThan18());

// שימוש
ValidationResult result = combined.apply(user);

מה זה אומר "extends Function"?

בג'אווה יש ממשק מובנה בשם Function<A, B> שמייצג פונקציה שמקבלת A ומחזירה B. הממשק שלנו מרחיב אותו: 

@FunctionalInterface
public interface UserValidation extends Function<User, ValidationResult> {
    // כלומר: UserValidation היא פונקציה  User → ValidationResult
}

המשמעות: כל UserValidation ניתנת לשימוש בכל מקום שמצפה ל-Function. אפשר גם להגדיר אחת בתור lambda — ביטוי פונקציונלי קצר: 

UserValidation myRule = user -> user.getAge() > 18
    ? new Valid()
    : new Invalid("גיל מינימלי 18");

כל מתודות השילוב (Combinators)

מתודה סוג מה היא עושה דוגמה
.and(other) instance שני הכללים חייבים לעבור (AND לוגי). Short-circuit — אם הראשון נכשל, השני לא נבדק A.and(B)
.or(other) instance מספיק שאחד יעבור (OR לוגי). Short-circuit — אם הראשון עובר, השני לא נבדק A.or(B)
.xor(other) instance בדיוק אחד חייב לעבור (XOR לוגי). שניהם נבדקים תמיד A.xor(B)
all(v1, v2, ...) static כל הכללים שהועברו חייבים לעבור. Varargs — מספר בלתי מוגבל של כללים all(A,B,C,D)
none(v1, v2, ...) static אף כלל לא יעבור. Varargs — מספר בלתי מוגבל של כללים none(A,B,C)

טבלת אמת — AND / OR / XOR

כלל Aכלל BA.and(B)A.or(B)A.xor(B)
✅ עובר✅ עובר✅✅❌
✅ עובר❌ נכשל❌✅✅
❌ נכשל✅ עובר❌✅✅
❌ נכשל❌ נכשל❌❌❌

4תבנית Factory Method

FACTORY METHOD PATTERN

מפעל — בונה אובייקטים בלי שתצטרך לדעת איזה

במקום שהקוד יבנה ישירות סוגים שונים של אובייקטים, הוא מבקש מ"מפעל" שייצר אותם.

ניקוד בסיכון: 10 נקודות

הבעיה שהתבנית פותרת

יש שלושה סוגי משתמשים: Basic, Premium, Platinum. בלי Factory, בכל מקום בקוד שרוצים ליצור משתמש כותבים: 

// ✗ בלי Factory — הקוד קשור ישירות לשמות המחלקות
User u1 = new BasicUser(...);
User u2 = new PremiumUser(...);

// אם מוסיפים DiamondUser, צריך לשנות בכל מקום בקוד

עם Factory, כל היצירה מרוכזת במקום אחד:

// ✓ עם Factory — קוד קורא לא יודע ולא צריך לדעת
User u = UserFactory.create("premium", "john", "john@test.co.il", "Pass$1", 25);

המימוש

public class UserFactory {

    public static User create(String type, String username,
                               String email, String password, int age) {
        // ה-Factory Method — מחליט איזה תת-מחלקה ליצור
        switch (type.toLowerCase()) {
            case "basic":    return new BasicUser(username, email, password, age);
            case "premium":  return new PremiumUser(username, email, password, age);
            case "platinum": return new PlatinumUser(username, email, password, age);
            default: throw new IllegalArgumentException("Unknown type: " + type);
        }
    }
}

5תבנית Template Method

TEMPLATE METHOD PATTERN

תבנית — שלד קבוע, פרט אחד מתחלף

מגדירים שלד אלגוריתם שחלק ממנו "פתוח" — הקורא ממלא את החלק המתחלף.

ניקוד בסיכון: 10 נקודות

הבעיה שהתבנית פותרת

רוצים למיין מערך של משתמשים. אלגוריתם המיון (bubble sort) תמיד אותו דבר — אבל הקריטריון לפיו ממיינים משתנה: פעם לפי גיל, פעם לפי שם, פעם לפי אימייל.

בלי Template Method היינו צריכים לכתוב sortByAge, sortByName, sortByEmail — קוד כפול עם לוגיקת מיון זהה.

המימוש

public static void sort(User[] users, Comparator<User> comparator) {
    int n = users.length;

    // שלד האלגוריתם (bubble sort) — קבוע, לא משתנה
    for (int i = 0; i < n - 1; i++) {
        for (int j = 0; j < n - i - 1; j++) {

            // ← זה ה"וו" (hook): הצעד המתחלף — מי גדול ממי?
            if (comparator.compare(users[j], users[j + 1]) > 0) {
                User temp  = users[j];
                users[j]   = users[j + 1];
                users[j + 1] = temp;
            }
        }
    }
}

כך משתמשים בו

// אותה מתודה, שלושה כיוונים שונים:

// מיון לפי גיל עולה
UserUtils.sort(users, (a, b) -> Integer.compare(a.getAge(), b.getAge()));

// מיון לפי שם משתמש אלפביתי
UserUtils.sort(users, (a, b) -> a.getUsername().compareTo(b.getUsername()));

// מיון לפי גיל יורד
UserUtils.sort(users, (a, b) -> Integer.compare(b.getAge(), a.getAge()));

6המחלקות והממשקים — מה בנינו

User — מחלקת הבסיס

מייצגת משתמש עם 4 שדות. הקונסטרקטור קורא ל-setters (לא מציב ישירות לשדות):

public User(String username, String email, String password, int age) {
    // דרך setters — לא this.username = username
    setUsername(username);
    setEmail(email);
    setPassword(password);
    setAge(age);
}

ValidationResult — ממשק התוצאה

כל בדיקת ולידציה מחזירה אובייקט מסוג זה. שני מימושים קונקרטיים:

ValidInvalid
isValid() מחזיר true תמיד מחזיר false תמיד
getReason() Optional.empty() — אין סיבה לכישלון Optional.of("סיבה") — מכיל הסבר
📌
למה Optional<String> ולא String?
Optional מאפשר לייצג "יש ערך" או "אין ערך" בלי להשתמש ב-null. כאשר הבדיקה עברה בהצלחה — אין סיבת כישלון, ולכן מחזירים Optional.empty() במקום null שעלול לגרום ל-NullPointerException.

UserValidation — ממשק הולידציה

זה הממשק המרכזי — Functional Interface שמרחיב Function<User, ValidationResult>. המשמעות: ניתן לייצג כלל בדיקה בתור lambda — ביטוי פונקציה קצר.

✅
עיקרון חשוב מאוד מה-Rejects:
כשמחזיקים reference לאובייקט, הטיפוס של המשתנה חייב להיות הממשק, לא המחלקה הקונקרטית:

Valid result = new Valid(); ← שגוי
ValidationResult result = new Valid(); ← נכון

8איך הכל עובד יחד — תהליך מלא

קוד הבדיקה שסיפקה המטלה:

public class UserValidationV2Demo {
    public static void main(String[] args) {

        // 1. יוצרים משתמש
        User user = new User("admin", "admin@#yzw.co.il", "abc123", 34);

        // 2. מגדירים שני כללים בנפרד
        UserValidation validation1 = UserValidation.emailLengthBiggerThan10();
        UserValidation validation2 = UserValidation.emailEndsWithIL();

        // 3. משלבים עם and ומפעילים
        ValidationResult result = (validation1.and(validation2)).apply(user);

        // 4. בודקים תוצאה
        if (result.isValid()) {
            System.out.println("User is valid");   // ← זה מודפס
        } else {
            System.out.println("User is not valid");
        }
    }
}

מה קורה בפנים שלב אחר שלב

יצירת המשתמש new User("admin", "admin@#yzw.co.il", "abc123", 34) קורא לקונסטרקטור שמפנה ל-setUsername, setEmail, setPassword, setAge.
יצירת הכלל הראשון UserValidation.emailLengthBiggerThan10() מחזיר lambda: user → user.getEmail().length() > 10 ? new Valid() : new Invalid(...)
יצירת הכלל השני UserValidation.emailEndsWithIL() מחזיר lambda: user → user.getEmail().endsWith("il") ? new Valid() : new Invalid(...)
שילוב עם and validation1.and(validation2) מחזיר כלל חדש שמפעיל את הראשון ואם עבר — גם את השני.
הפעלה על המשתמש .apply(user) — האימייל "admin@#yzw.co.il" ארוך 17 תווים (>10 ✓) ומסתיים ב-il ✓ → מחזיר new Valid()
בדיקת התוצאה result.isValid() מחזיר true → מודפס "User is valid"
